# D模块完成总结

## 1. 模块职责

我负责的 D 模块主要是 ANN 索引部分，核心职责包括：

- 基于已入库的 `CellVector` 构建 ANN 索引
- 支持多种索引类型的创建与校验
- 生成并落盘索引产物
- 维护索引版本、发布和回滚流程
- 支持索引对象的进程内加载与缓存
- 为检索模块提供稳定一致的向量顺序和索引映射

当前实现是基于现有后端工程 `ANN-retrieval-system/backend` 继续补充，而不是单独拆出一套索引服务。这样可以直接接入现有的数据预处理、数据库、任务调度和检索链路。

## 3. 当前索引输入的实际结论

根据当前数据链路，D 模块使用的索引输入不是原始 `.h5ad`，而是已经预处理入库后的 `CellVector`。

当前约定如下：

- 索引输入来源：`CellVector`
- 索引主向量：`vector_type = "pca"`
- 向量维度：`30`
- 向量顺序：按 `CellVector.id.asc()` 读取
- `cell_id` 与 `CellVector.cell_id` 保持一一映射

因此，D 模块不需要自己重新解析 `.h5ad`，只需要从数据库读取已准备好的 PCA 向量即可。

## 4. 具体改动说明

### 4.1 `index_tasks.py`

文件：

- `ANN-retrieval-system/backend/app/tasks/index_tasks.py`

主要改动如下：

- 构建索引任务从数据库读取 `CellVector`
- 读取时固定过滤条件：
  - `dataset_id`
  - `vector_type = "pca"`
- 读取结果增加 `order_by(CellVector.id.asc())`
- 使用统一的向量矩阵转换逻辑生成训练数据
- 构建完成后写回任务状态、索引状态、召回率和产物大小

这一部分确保了索引构建阶段的输入顺序和数据来源都稳定可控。

### 4.2 `ann_engine.py`

文件：

- `ANN-retrieval-system/backend/app/services/ann_engine.py`

主要能力如下：

- 统一提供向量矩阵转换和合法性检查
- 校验索引配置是否满足构建条件
- 支持 `flat`、`ivf_pq`、`hnsw` 三种索引构建
- 支持 `l2`、`ip`、`cosine` 三种 metric
- 生成索引映射文件 `id_map`
- 生成索引元数据文件 `meta`
- 记录构建耗时和产物大小
- 支持运行时索引加载

其中，`id_map.json` 用于保存内部向量位置和数据库记录之间的映射，`meta.json` 用于保存索引版本、维度、参数和产物路径等信息。

### 4.3 `index_service.py`

文件：

- `ANN-retrieval-system/backend/app/services/index_service.py`

主要改动如下：

- 在创建索引任务前先做数据和权限检查
- 在任务创建时校验索引参数合法性
- 在构建前检查当前数据集是否存在可用 PCA 向量
- 在加载索引时支持进程内缓存
- 支持索引发布与回滚
- 支持缓存失效，避免旧索引对象继续被使用

这部分把索引从“能构建”补到了“能管理、能发布、能回滚、能加载”。

## 6. 需要和其他同学说明清楚的内容

### 6.1 给 C 同学的说明

- D 模块不直接处理原始 `.h5ad`
- D 模块只消费已经入库的 `CellVector`
- 索引主向量固定使用 `vector_type = "pca"`
- 索引构建时必须保证向量顺序稳定

### 6.2 给 E 同学的说明

- E 模块可以直接依赖 D 模块生成的索引产物进行检索
- 检索时需要使用同一套向量顺序和 `id_map`
- 不能自行改变内部向量与 `cell_id` 的映射关系
- 如果使用加载到内存的索引对象，需要注意发布和回滚后的缓存失效

### 6.3 给后续联调同学的说明

- 不要跳过索引参数校验直接构建
- 不要自行重排 `CellVector` 的读取顺序
- 不要把 `UMAP`、`TSNE` 当作 ANN 主向量
- `id_map.json` 和 `meta.json` 是索引运行和追踪的重要产物，不能省略
